# 深度研究

研究主题：$ARGUMENTS

## 何时使用（vs 网页搜索）

仅当用户明确要求进行深度/详尽研究时才使用此功能。深度研究比网页搜索慢 10-100 倍且成本更高。对于普通的“研究 X”请求、快速查找或事实核查，请改用 **网页搜索**。

## 步骤 1：启动研究

将研究目标框定为优先考虑学术文献。如果用户的查询是科学或技术性的，则在参数前添加上下文，以引导向学术来源 —— 例如，不要用 `"睡眠剥夺的影响"`，而用 `"关于睡眠剥夺影响的同行评审研究和临床研究"`。这样可以在不排除相关非学术来源的情况下，将研究代理引向一手文献。

```bash
parallel-cli research run "$ARGUMENTS" --processor pro-fast --no-wait --json
```

如果这是对之前已知 `interaction_id` 的研究或丰富任务的**后续跟进**，则添加上下文链：

```bash
parallel-cli research run "$ARGUMENTS" --processor lite --no-wait --json --previous-interaction-id "$INTERACTION_ID"
```

通过跨请求链接 `interaction_id` 值，每个后续问题都会自动包含前序对话的完整上下文 —— 这样你就可以更深入地挖掘一个主题，而无需重申已研究过的内容。后续跟进使用 `--processor lite`，因为繁重的研究已经在初始轮次中完成，后续只需在此基础上继续。

该命令会立即返回。**切勿**省略 `--no-wait` —— 没有它，命令将阻塞几分钟并最终超时。

处理器选项（根据用户请求选择）：

| 处理器 | 预期延迟 | 使用场景 |
|--------|----------|----------|
| `pro-fast` | 30 秒 - 5 分钟 | 默认 —— 深度和速度的良好平衡 |
| `ultra-fast` | 1 - 10 分钟 | 更深入分析，更多来源（约 2 倍成本） |
| `ultra` | 5 - 25 分钟 | 最大深度，仅在明确要求时使用（约 3 倍成本） |

解析 JSON 输出以提取 `run_id`、`interaction_id` 和监控 URL。立即告知用户：
- 深度研究已启动
- 所选处理器级别的预期延迟（见上表）
- 他们可以跟踪进度的监控 URL

告诉他们可以在后台执行轮询步骤，以便在研究运行时继续工作。

## 步骤 2：轮询获取结果

根据主题选择一个描述性文件名（例如 `ai-chip-market-2026`、`react-vs-vue-comparison`）。使用小写字母和连字符，不含空格。

```bash
parallel-cli research poll "$RUN_ID" -o "$FILENAME" --timeout 540
```

重要：
- 使用 `--timeout 540`（9 分钟）以保持在工具执行限制内
- **不要**传递 `--json` —— 完整输出很大，会淹没上下文。`-o` 标志会将结果写入文件
- `-o` 标志会生成两个输出文件：
  - `$FILENAME.json` —— 元数据和依据
  - `$FILENAME.md` —— 格式化 markdown 报告
- 轮询命令在研究完成时会在 stdout 打印**执行摘要**。将此执行摘要分享给用户 —— 它让用户无需打开文件即可快速了解概况。

### 如果轮询超时

更高级别的处理器可能需要超过 9 分钟。如果轮询在未完成时退出：
1. 告知用户研究仍在服务器端运行
2. 重新运行相同的 `parallel-cli research poll` 命令以继续等待

## 响应格式

**步骤 1 后：** 分享监控 URL（仅用于跟踪进度 —— 并非最终报告）。

**步骤 2 后：**
1. 分享轮询命令在 stdout 打印的**执行摘要**
2. 简要评估来源质量：引用的来源中有多少是同行评审期刊、预印本或机构报告，而有多少是新闻文章或博客文章。如果学术覆盖薄弱，请向用户指出，并建议针对特定数据库进行后续检索（例如，“我可以专门在 PubMed/arXiv 上搜索此主题的更多一手来源”）。
3. 告诉用户生成的两个文件路径：
   - `$FILENAME.md` —— 格式化 markdown 报告
   - `$FILENAME.json` —— 元数据和依据
4. 分享 `interaction_id` 并告知用户他们可以提出基于此研究的后续问题（例如，“更深入地研究 X” 或 “将其与 Y 进行比较”）

完成之后**不要**再次分享监控 URL —— 结果在文件中，而不在该链接。

询问用户是否希望阅读文件以了解详情。除非用户要求，否则不要将文件内容读入上下文。

**记住 `interaction_id`** —— 如果用户提出与此研究相关的后续问题，请将其作为 `--previous-interaction-id` 用在下一个研究或丰富命令中。

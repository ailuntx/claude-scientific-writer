# ML会议论文写作指南

NeurIPS、ICML、ICLR、CVPR、ECCV、ICCV及其他主要机器学习和计算机视觉会议的综合性写作指南。

**最后更新**：2024年

---

## 概述

ML会议高度重视**创新性**、**严格的实证评估**和**可复现性**。论文评估标准包括：贡献明确、基线强大、 ablation全面、局限性讨论诚实。

### 核心理念

> "展示而非叙述——你的实验应该证明你的论点，而不仅仅是你的文字。"

**主要目标**：通过严格实验验证的新方法推动技术进步。

---

## 读者群体与文风

### 目标读者

- ML研究人员和从业者
- 特定子领域的专家
- 熟悉近期文献
- 期望技术深度和精确性

### 文风特点

| 特点 | 描述 |
|---------------|-------------|
| **技术性** | 方法论细节密集 |
| **精确** | 术语精确，无歧义 |
| **实证** | 主张有实验支撑 |
| **直接** | 贡献阐述清晰 |
| **诚实** | 承认局限性 |

### 语气

- **第一人称复数（"我们"）**："我们提出..." "我们的方法..."
- **主动语态**："我们引入了一种新的架构..."
- **自信但适度**：强主张需要强证据。

## 摘要

### 风格要求

- **密集且聚焦于数字**
- **150-250词**（因会议而异）
- **关键结果置前**：包含具体指标
- **流畅段落**（非结构化）

### 摘要结构

1. **问题**（1句）：你在解决什么问题？
2. **现有工作的局限性**（1句）：当前方法为何不足
3. **你的方法**（1-2句）：你的方法是什么？
4. **关键结果**（2-3句）：基准测试的具体数字
5. **意义**（可选，1句）：这为何重要

### 示例摘要（NeurIPS风格）

```
Transformers在序列建模方面取得了显著成功，但存在二次计算复杂度，
限制了其在长序列中的应用。我们引入了FlashAttention-2，一种IO感知的精确
注意力算法，相比FlashAttention实现2倍加速，相比标准注意力在最长16K
令牌的序列上实现最高9倍加速。我们的关键洞察是通过分块和重计算减少
内存读写，实现最优IO复杂度。在Long Range Arena基准测试上，
FlashAttention-2支持8倍更长的序列训练，同时保持标准注意力的精度。
结合序列并行，我们在64K令牌的序列上以近线性成本训练GPT风格模型。
我们发布了优化的CUDA内核，在A100 GPU上实现80%的理论峰值FLOPS。
代码见[anonymous URL]。
```

### 摘要禁忌

❌ "我们提出了一种X的新方法"（模糊，无结果）
❌ "我们的方法优于基线"（无具体数字）
❌ "这是一个重要问题"（不言自明的主张）

✅ 包含具体指标："达到94.5%准确率，提升3.2%"
✅ 包含规模："在100万样本上"或"16K令牌序列"
✅ 包含对比："比之前的SOTA快2倍"

## 引言

### 结构（2-3页）

ML引言具有独特的结构，包含**编号的贡献列表**。

### 逐段指南

**第一段：问题动机**
- 为什么这个问题重要？
- 应用场景是什么？
- 设定技术挑战

```
"大型语言模型在自然语言理解和生成方面展示了显著能力。然而，
二次注意力复杂度成为处理长文档、多轮对话和扩展上下文推理的
根本瓶颈。随着模型扩展到数十亿参数，上下文长度延伸到数万令牌，
高效的注意力机制对于实际部署变得至关重要。"
```

**第二段：现有方法的局限性**
- 存在哪些方法？
- 为什么它们不够？
- 局限性的技术分析

```
"先前的工作通过稀疏注意力模式、线性注意力近似和低秩分解来解决
这一问题。虽然这些方法降低了理论复杂度，但它们通常牺牲精度，
需要专用硬件，或引入在深度网络中累积的近似误差。精确注意力
在计算资源允许时仍是首选。"
```

**第三段：你的方法（高层）**
- 你的关键洞察是什么？
- 你的方法概念上如何工作？
- 为什么它应该成功？

```
"我们观察到注意力中的主要瓶颈不是计算而是内存带宽——在现代
GPU上读写大型N×N注意力矩阵占主导了运行时间。我们提出
FlashAttention-2，通过一种新的分块策略消除这一瓶颈，逐块
计算注意力而无需实例化完整矩阵。"
```

**第四段：贡献列表（关键）**

这在ML会议中是**必需的且独特的**：

```
我们的贡献如下：

• 我们提出了FlashAttention-2，一种IO感知的精确注意力算法，
  实现最优内存复杂度O(N²d/M)，其中M是GPU SRAM大小。

• 我们提供了理论分析，表明我们的算法在典型GPU配置上
  比FlashAttention少2-4倍的HBM访问。

• 我们展示了在256到64K令牌的序列长度上比FlashAttention
  快2倍，比标准PyTorch注意力快最多9倍。

• 我们表明FlashAttention-2支持在同一硬件上8倍更长的上下文
  训练，为长程建模解锁新能力。

• 我们在[anonymous URL]发布了优化的CUDA内核和PyTorch绑定。
```

### 贡献要点指南

| 好的贡献要点 | 差的贡献要点 |
|--------------------------|-------------------------|
| 具体、可量化 | 模糊的主张 |
| 自包含 | 需要阅读论文才能理解 |
| 彼此独立 | 重复的要点 |
| 强调创新性 | 陈述显而易见的事实 |

### 相关工作位置

- **引言中**：简要定位（1-2段）
- **单独章节**：详细比较（结尾或结论前）
- **附录**：空间受限时扩展讨论

## 方法

### 结构（2-3页）

```
方法
├── 问题公式化
├── 方法概述/架构
├── 关键技术组件
│   ├── 组件1（含方程）
│   ├── 组件2（含方程）
│   └── 组件3（含方程）
├── 理论分析（如适用）
└── 实现细节
```

### 数学符号

- **定义所有符号**："设X ∈ ℝ^{N×d}表示输入序列..."
- **符号一致**：同一符号始终表示相同含义
- **编号重要方程**：后续按编号引用

### 算法伪代码

为确保可复现性，包含清晰的伪代码：

```
算法1：FlashAttention-2前向传播
─────────────────────────────────────────
输入：Q, K, V ∈ ℝ^{N×d}, 块大小B_r, B_c
输出：O ∈ ℝ^{N×d}

1:  将Q分为T_r = ⌈N/B_r⌉个块
2:  将K, V分为T_c = ⌈N/B_c⌉个块
3:  初始化O = 0, ℓ = 0, m = -∞
4:  for i = 1 to T_r do
5:    将Q_i从HBM加载到SRAM
6:    for j = 1 to T_c do
7:      将K_j, V_j从HBM加载到SRAM
8:      计算S_ij = Q_i K_j^T
9:      更新运行最大值和求和
10:     增量更新O_i
11:   end for
12:   将O_i写入HBM
13: end for
14: return O
```

### 架构图

- **清晰、出版质量的图表**
- **标注所有组件**
- **用箭头显示数据流**
- **使用一致的视觉语言**

## 实验

### 结构（2-3页）

```
实验
├── 实验设置
│   ├── 数据集和基准
│   ├── 基线
│   ├── 实现细节
│   └── 评估指标
├── 主要结果
│   └── 包含主要比较的表格/图
├── 消融研究
│   └── 组件分析
├── 分析
│   ├── 缩放行为
│   ├── 定性示例
│   └── 错误分析
└── 计算效率
```

### 数据集和基准

- **使用标准基准**：确保可比性
- **报告数据集统计**：大小、划分、预处理
- **说明非标准选择的原因**：如果使用自定义数据，解释原因

### 基线

**对接收至关重要。** 包含：
- **近期SOTA**：不仅是旧方法
- **公平比较**：相同计算预算、超参数调优
- **消融版本**：你的方法去掉关键组件
- **强大基线**：不要 cherry-pick 弱竞争者

### 主要结果表

清晰、全面的格式：

```
表1：Long Range Arena基准测试结果（准确率%）
──────────────────────────────────────────────────────────
方法           | ListOps | 文本 | 检索 | 图像 | 路径 | 平均
──────────────────────────────────────────────────────────
Transformer    |  36.4   | 64.3 |  57.5  | 42.4 | 71.4 | 54.4
Performer      |  18.0   | 65.4 |  53.8  | 42.8 | 77.1 | 51.4
Linear Attn    |  16.1   | 65.9 |  53.1  | 42.3 | 75.3 | 50.5
FlashAttention |  37.1   | 64.5 |  57.8  | 42.7 | 71.2 | 54.7
FlashAttn-2    |  37.4   | 64.7 |  58.2  | 42.9 | 71.8 | 55.0
──────────────────────────────────────────────────────────
```

### 消融研究（必需）

展示你的方法中什么重要：

```
表2：FlashAttention-2组件消融研究
──────────────────────────────────────────────────────────
变体                               | 加速 | 内存
──────────────────────────────────────────────────────────
完整FlashAttention-2               |  2.0x | 1.0x
  - 无序列并行                     |  1.7x | 1.0x
  - 无重计算                       |  1.3x | 2.4x
  - 无块分块                       |  1.0x | 4.0x
FlashAttention-1（基线）            |  1.0x | 1.0x
──────────────────────────────────────────────────────────
```

### 消融应展示什么

- **每个组件都重要**：去掉它会损害性能
- **设计选择合理**：为什么是这个架构/超参数？
- **失败模式**：方法何时不工作？
- **敏感性分析**：对超参数的鲁棒性

## 相关工作

### 位置选项

1. **引言之后**：CV论文中常见
2. **结论之前**：NeurIPS/ICML中常见
3. **附录**：空间紧张时

### 写作风格

- **按主题组织**：而非按时间顺序
- **定位你的工作**：你与每条工作线的区别
- **公平描述**：不要歪曲先前工作
- **近期引用**：包含2023-2024年的论文

### 示例结构

**高效注意力机制。** 先前关于高效注意力的工作分为三类：稀疏模式
（Beltagy et al., 2020; Zaheer et al., 2020）、线性近似（Katharopoulos
et al., 2020; Choromanski et al., 2021）和低秩分解（Wang et al., 2020）。
我们的工作不同之处在于关注IO高效的精确注意力而非近似。

**内存高效训练。** 梯度检查点（Chen et al., 2016）和激活重计算
（Korthikanti et al., 2022）通过牺牲计算来减少内存。我们采用类似
想法，但将其应用于注意力算子本身。

## 局限性章节

### 为什么重要

NeurIPS、ICML、ICLR**越来越要求**。诚实的局限性：
- 展示科学成熟度
- 指导未来工作
- 防止过度宣传

### 应包含的内容

1. **方法局限性**：何时失败？
2. **实验局限性**：什么未测试？
3. **范围局限性**：什么不在范围内？
4. **计算局限性**：资源需求

### 示例局限性章节

**局限性。** 虽然FlashAttention-2提供了显著加速，但仍存在几个局限。
首先，我们的实现针对NVIDIA GPU优化，不支持AMD或其他硬件。其次，
加速在中等长度到长序列上最为明显；对于非常短的序列（<256令牌），
内核启动开销占主导。第三，我们专注于密集注意力；将我们的方法
扩展到稀疏注意力模式仍是未来工作。最后，我们的理论分析假设特定
的GPU内存层次结构参数，可能不适用于未来的硬件代际。

## 可复现性

### 可复现性清单（NeurIPS/ICML）

大多数ML会议要求涵盖以下内容的可复现性清单：

- [ ] 代码可用性
- [ ] 数据集可用性
- [ ] 超参数已指定
- [ ] 随机种子已报告
- [ ] 计算需求已说明
- [ ] 运行次数和方差已报告
- [ ] 统计显著性检验

### 应报告的内容

**超参数**：
```
"我们使用Adam（β₁=0.9, β₂=0.999, ε=1e-8）和3e-4的学习率，
在1000步内线性预热后余弦退火。批量大小为256，跨8个A100 GPU。
我们训练100K步（约24小时）。"
```

**随机种子**：
```
"所有实验在3个随机种子（0, 1, 2）上取平均，标准差在括号中报告。"
```

**计算**：
```
"实验在8个NVIDIA A100-80GB GPU上进行。总训练时间约500 GPU小时。"
```

## 图表

### 图表质量

- **优先使用矢量图形**：PDF, SVG
- **光栅图高分辨率**：300+ dpi
- **出版尺寸下可读**：在实际栏宽下测试
- **色盲友好**：除颜色外使用图案

### 常见图表类型

1. **架构图**：视觉展示你的方法
2. **性能图**：学习曲线、缩放行为
3. **比较表**：主要结果
4. **消融图**：组件贡献
5. **定性示例**：输入/输出样本

### 图表说明

自包含的说明，解释：
- 显示了什么
- 如何阅读图表
- 关键要点

## 参考文献

### 引用风格

- **编号[1]** 或 **作者-年份（Smith et al., 2023）**
- 检查会议特定要求
- 全篇保持一致

### 参考文献指南

- **引用近期工作**：期望2022-2024年的论文
- **不要过度引用自己**：会引起偏见担忧
- **适当引用arxiv**：可用时使用已发表版本
- **包含所有相关先前工作**：缺失引用会损害审稿

## 会议特定说明

### NeurIPS

- **8页**正文 + 无限附录/参考文献
- **更广泛影响**章节有时必需
- **可复现性清单**必需
- OpenReview提交，公开评审

### ICML

- **8页**正文 + 无限附录/参考文献
- 强调整论**理论 + 实验**
- 鼓励可复现性声明

### ICLR

- **8页**正文（最终版可超页）
- OpenReview及**公开评审和讨论**
- 作者回应期是交互式的
- 强调整**创新性和洞察**

### CVPR/ICCV/ECCV

- **正文含参考文献8页**
- **鼓励补充视频**
- 高度重视**视觉结果**
- 基准性能关键

## 常见错误

1. **基线薄弱**：未与近期SOTA比较
2. **缺失消融**：未展示组件贡献
3. **过度声称**：当你只部分解决X时说"我们解决X"
4. **贡献模糊**："我们提出了一种新方法"
5. **可复现性差**：缺失超参数、种子
6. **错误的模板**：使用去年的样式文件
7. **匿名违规**：在盲审中暴露身份
8. **缺失局限性**：未承认失败模式

## 反驳技巧

ML会议有作者回应期。技巧：
- **首先解决关键问题**：优先处理关键问题
- **运行要求的实验**：时间允许时
- **简洁**：审稿人阅读很多反驳
- **保持专业**：即使面对不公平的评审
- **引用具体行**：如"L127所述..."
